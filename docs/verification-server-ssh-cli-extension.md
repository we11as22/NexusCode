# Верификация: сервер / не-сервер, SSH / не-SSH, CLI и расширение

Проверка, что весь функционал, работа с файлами и все тулзы полноценно работают во всех режимах.

---

## 1. Режимы работы

| Режим | Описание |
|-------|----------|
| **CLI без сервера** | `nexus` без `--server` / без `NEXUS_SERVER_URL`. Агент в процессе, `CliHost`, сессии в `~/.nexus/sessions/{hash}/`. |
| **CLI с сервером** | `nexus --server URL` или `NEXUS_SERVER_URL=URL`. Сессии и запуск на сервере, стрим событий в CLI; при `tool_end` (write_to_file/replace_in_file) CLI дописывает файлы локально по `cwd` и `event.path`. |
| **Расширение без сервера** | В настройках **Server URL** пусто. Агент в процессе расширения, `VsCodeHost`, сессии локально. |
| **Расширение с сервером** | **Server URL** задан (например `http://127.0.0.1:4097`). Создание/стрим через POST `/session/:id/message`; при `tool_end` с `writtenContent` расширение пишет в workspace по `cwd` и `event.path`. |
| **SSH / удалённый сервер** | Сервер запущен на другой машине (в т.ч. по SSH). Клиент (CLI или расширение) подключается по URL. Важно: `directory` (cwd) — путь **на сервере**; файловые операции выполняются на сервере. Клиент при получении `tool_end` с `writtenContent` пишет в **свой** `cwd` (локальный или workspace). Для корректной работы путей при удалённом сервере см. раздел 4. |

---

## 2. Работа с файлами и тулзы по режимам

### 2.1 Чтение/запись файлов (read_file, write_to_file, replace_in_file)

- **Core:** все тулзы вызывают `ctx.host.readFile` / `ctx.host.writeFile`; путь резолвится относительно `host.cwd` (абсолютный остаётся как есть).
- **CLI без сервера:** `CliHost(cwd)` — `resolve(path)` = `path.resolve(this.cwd, path)`. Чтение/запись через `fs` на локальной ФС. Ограничения: `DENY_EXTENSIONS`, `DENY_PATHS` (в т.ч. `.env`, `.ssh`).
- **CLI с сервером:** Сервер выполняет все файловые операции в своей `cwd` (из `x-nexus-directory`). В ответе стримятся события `tool_end` с `path` и `writtenContent`. CLI в `index.ts` при получении такого события дописывает файл локально: `path.join(cwd, event.path)` и `fs.writeFile` — т.е. ожидается, что локальный `cwd` соответствует проекту (тот же путь или синхронизированная копия).
- **Расширение без сервера:** `VsCodeHost(cwd)` — `readFile`/`writeFile` через `vscode.workspace.fs`, пути через `path.join(this.cwd, filePath)` (cwd = workspace folder).
- **Расширение с сервером:** Стрим от сервера; при `tool_end` с `write_to_file`/`replace_in_file` и `writtenContent` расширение создаёт директории при необходимости и пишет в `vscode.Uri.file(path.join(cwd, event.path))`, где `cwd` — текущий workspace. Запись идёт в workspace расширения (локальный или Remote-SSH).

Итог: во всех четырёх комбинациях (CLI/Extension × server/no server) запись файлов идёт в нужное место при условии, что `cwd` и переданный в запросе `directory` указывают на один и тот же проект (или синхронизированную копию).

### 2.2 Выполнение команд (execute_command)

- **Core:** `ctx.host.runCommand(command, ctx.cwd, ctx.signal)`.
- **CliHost:** `execa(command, { shell: true, cwd, timeout: 120_000, signal })` — добавлена передача `signal` для отмены по Abort (Ctrl+C).
- **ServerHost:** `execa(..., { signal })` — отмена по закрытию стрима/abort.
- **VsCodeHost:** `execa(..., { signal })` — отмена по abort.

Во всех режимах команда выполняется в переданном `cwd`; при использовании сервера — на машине сервера.

### 2.3 Остальные тулзы (grep, codebase_search, list_files, list_code_definitions, MCP, spawn_agent и т.д.)

- Используют тот же `ctx.host` (readFile, runCommand и т.д.) и `ctx.cwd`.
- В режиме с сервером все они выполняются **на сервере** в его `cwd`; индекс (symbol/vector) строится на сервере для этой `cwd`.
- В режиме без сервера — в процессе CLI или расширения, индекс (если включён) локальный.

## 3. Сессии и сообщения

- **Без сервера:** сессии в файловой системе: `~/.nexus/sessions/{projectHash}/{sessionId}.jsonl`; сообщения и todo в файле.
- **С сервером:** сессии и сообщения в БД сервера (SQLite); список — GET `/session?directory=`, сообщения — GET `/session/:id/message` с пагинацией. CLI и расширение при `serverUrl` не пишут сессии на диск; переключение сессий идёт через API.

---

## 4. SSH и удалённый сервер

- Явной поддержки SSH в коде нет: сервер — обычный HTTP (Hono); подключение по URL (в т.ч. `http://remote-host:4097` или через SSH-туннель).
- При запуске сервера на удалённой машине:
  - **directory** в запросах — путь **на сервере** (например `/home/user/project`). Все файловые операции и команды выполняются там.
  - **CLI:** при использовании `--server` и `--project /path` файлы по событиям `tool_end` пишутся в **локальный** `/path`. Для согласованности при удалённом сервере обычно запускают CLI **на той же машине**, где проект (например, заходят по SSH и там выполняют `nexus --server http://127.0.0.1:4097` с локальным `cwd`).
  - **Расширение:** при Remote-SSH workspace уже на удалённой машине, `getCwd()` — удалённый путь; если сервер тоже на этой машине и слушает 127.0.0.1, то и сервер, и запись из событий работают с одной и той же ФС.

---

## 5. Проверенные исправления (в коде)

- **AbortSignal в runCommand:** в `CliHost` и `VsCodeHost` в `runCommand` в `execa` передаётся `signal`, чтобы по отмене (Ctrl+C / abort) прерывалось и выполнение команды.
- **Сервер:** `ServerHost` уже передавал `signal` в `execa`.

---

## 6. Краткий чеклист проверки

| Сценарий | Ожидание |
|----------|----------|
| CLI, без сервера: read_file, write_to_file, replace_in_file | Файлы в `cwd` (текущая или `--project` директория). |
| CLI, без сервера: execute_command | Команда в `cwd`, по Ctrl+C команда прерывается. |
| CLI, с сервером: отправка сообщения, стрим | События приходят; при write_to_file/replace_in_file файл появляется/обновляется в локальном `cwd`. |
| Расширение, без сервера: те же тулзы | Файлы в workspace; команды в cwd; отмена прерывает runCommand. |
| Расширение, с сервером: отправка сообщения, стрим | События в UI; при tool_end с writtenContent файл в workspace обновляется. |
| Сервер: запуск агента с directory | Агент работает в переданной `cwd`; все host-операции на сервере. |
| Переключение сессий (CLI/расширение) с сервером | Список сессий с сервера; загрузка последних сообщений по offset/limit. |

При необходимости можно добавить автоматические тесты на основе этого чеклиста (E2E для CLI/extension и интеграционные для server).
