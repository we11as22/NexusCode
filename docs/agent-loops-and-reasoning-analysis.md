# Комплексный анализ: агентные лупы, ризонинг в логике и схемах, связь со встроенным ризонингом модели

## 1. Структура агентной лупы по проектам

### 1.1 NexusCode (наш проект)

**Где:** `packages/core/src/agent/loop.ts` — `runAgentLoop()`.

**Как устроена лупа:**
- Один внешний `while (true)`: на каждой итерации — сбор системного промпта, сообщений (`buildMessagesFromSession`), списка тулов (built-in + dynamic/MCP), один вызов `client.stream({ messages, tools, systemPrompt, ... })`.
- Обработка стрима: события `text_delta`, `reasoning_delta`, `tool_call`, `finish`. Текст и reasoning накапливаются в `currentText` / `currentReasoning`, при каждом изменении вызывается `flushAssistantContent()` — контент пишется в сообщение ассистента в сессии (порядок: reasoning, text, tool parts). При `tool_call` — создаётся tool part, при необходимости запрос на подтверждение, затем выполнение тула (сразу или батчем для read-only). Результаты тулов пишутся в сессию через `session.updateToolPart`.
- После окончания стрима: если были tool_calls — **не выходим из цикла**; следующая итерация снова собирает сообщения (уже с результатами тулов) и снова вызывает `stream()`. Выход из цикла: `attempt_completion` / `plan_exit`, `finishReason === "stop"` без тулов, аборт, превышение лимита итераций или тулов, фатальная ошибка. При достижении лимита итераций в системный промпт добавляется блок «MAXIMUM STEPS REACHED», тулы отключаются (`llmTools = []`), запрашивается только текст.

**Где агент «думает» / планирует:**
- **Встроенный ризонинг модели:** события `reasoning_delta` из провайдера (o1, R1, OpenRouter thinking и т.д.) накапливаются в `currentReasoning`, сохраняются в сообщении как `{ type: "reasoning", text }`. В `buildMessagesFromSession` при формировании контента ассистента **сначала** идут все блоки reasoning, потом текст, потом tool-call'ы — так модель в следующем запросе снова видит своё рассуждение. Отдельных параметров API под reasoning effort / thinking budget мы **не** передаём — только проброс стрима.
- **Тул `thinking_preamble`:** модель может явно вызвать тул с полями `reasoning_and_next_actions` (обязательно) и `user_message` (опционально). Это не нативный стрим, а **явная запись плана/рассуждений в контекст**; выполнение тула только проверяет «не два раза подряд» и возвращает успех. Текст рассуждений оказывается в контенте сообщения (вход тула), т.е. в следующем запросе модель видит свой же план. Отдельной фазы «сначала спланировать тулы, потом выполнить» в коде **нет** — модель сама решает, вызывать ли thinking_preamble перед батчем тулов.
- **Планирование тулов:** явного шага «сначала план тулов по схеме, потом выполнение» нет. Модель в одном ответе может выдать несколько tool_calls; мы их выполняем (с батчингом для read-only) и на следующей итерации отдаём результаты. Классификация динамических тулов по задаче (`classifyTools`) — это предварительный отбор списка тулов до лупы, а не «режим планирования».

**Ризонинг в схемах:**
- В параметрах тулов ризонинга нет (кроме `thinking_preamble`: поле `reasoning_and_next_actions` — по смыслу рассуждение). В классификаторах (`classifier.ts`) есть опциональное поле `reasoning` в ответе — используется для отбора тулов/скиллов, не для агентной лупы.

### 1.2 Cline

**Где:** `src/core/task/index.ts` (Task), стрим обрабатывается через `StreamResponseHandler`, выполнение тулов — `ToolExecutor` и хендлеры по типам тулов.

**Как устроена лупа:**
- Задача создаётся через `controller.initTask()`. Стрим от API приходит и разбирается в `StreamResponseHandler`: **ToolUseHandler** накапливает дельты tool_use (id, name, input), **ReasoningHandler** накапливает дельты reasoning (content, signature, details, redacted_data). По завершении стрима тулы финализируются и выполняются (через ToolExecutor и хендлеры, напр. `PlanModeRespondHandler`, `ActModeRespondHandler`). После выполнения тулов формируется следующий запрос к API (история + результаты тулов) — т.е. **один запрос = один «ответ» модели**, при наличии tool_calls выполнение идёт, затем новый запрос. Точное место, где решается «ещё один запрос или конец», находится в логике Task (обработка finish, ask и т.д.).
- **Plan mode vs Act mode:** в системном промпте задаётся режим. **PLAN MODE:** доступны только read-only тулы + **`plan_mode_respond`** (один тул для ответа пользователю и представления плана); цель — собрать контекст и написать план, пользователь потом переключает в ACT MODE. **ACT MODE:** все тулы кроме `plan_mode_respond`; для прогресс-сообщений без остановки цикла используется **`act_mode_respond`** (не блокирует). То есть «планирование» в Cline — это **режим с ограниченным набором тулов и одним тулом для вывода плана**, а не отдельная фаза «сначала план тулов в коде».

**Где агент «думает» / планирует:**
- **Встроенный ризонинг:** ReasoningHandler обрабатывает стрим reasoning (reasoning_content и т.п.), сохраняет в контент сообщения как блоки `type: "thinking"`. В `r1-format.ts` и в преобразованиях для провайдеров это мышление при необходимости прокидывается обратно в API (напр. `reasoning_content` для DeepSeek R1). В настройках есть **Reasoning effort** и **Thinking budget** (`planModeReasoningEffort`, `actModeReasoningEffort`, `planModeThinkingBudgetTokens`, `actModeThinkingBudgetTokens`) — они передаются в API провайдерам, т.е. управляют **встроенным** ризонингом модели, а не отдельным шагом в коде.
- **Планирование:** в PLAN MODE модель использует read-only тулы + `plan_mode_respond`, чтобы выдать план; в коде нет фазы «сначала сгенерировать план тулов по схеме». Промпт просит «think about which tool is most relevant… close the thinking tag and proceed with the tool use» — то есть использование тегов/встроенного мышления перед вызовом тула задаётся в промпте, не в архитектуре лупы.

**Ризонинг в схемах:**
- Схемы тулов в spec — параметры (name, type, required, instruction и т.д.); отдельного поля «reasoning» в схемах тулов нет. `plan_mode_respond` и `act_mode_respond` имеют параметры вроде `response`, `needs_more_exploration` — это текст ответа/плана, не отдельная схема ризонинга.

### 1.3 Kilocode / OpenCode

**Где:** `packages/opencode/src/session/prompt.ts` — цикл по шагам; на каждом шаге вызывается **`processor.process(streamInput)`** (`session/processor.ts`).

**Как устроена лупа:**
- В prompt.ts цикл `while (true)`: собираются tools, system, messages; при включённом structured output добавляется тул **StructuredOutput** и в системный промпт — инструкция использовать его, `toolChoice: "required"`. Затем **`processor.process({ messages, tools, system, toolChoice, ... })`**. Внутри process — **один** вызов `LLM.stream(streamInput)` и обход `stream.fullStream`.
- В **processor** обрабатываются события: `reasoning-start` → создаётся часть типа reasoning, пишется в сессию; `reasoning-delta` → обновление текста части; `reasoning-end` → финализация части. Затем `tool-input-start` / `tool-call` и т.д. — создаются части тулов, по завершении тулов они выполняются. Результат process возвращает управление в prompt.ts; если был вызов StructuredOutput — цикл завершается; иначе при `finish === "tool-calls"` цикл продолжается (следующий шаг с теми же сообщениями + результаты тулов). Есть лимит шагов; при достижении в сообщения добавляется MAX_STEPS (аналог нашего «tools disabled»).
- **План:** есть тулы **plan_enter** и **plan_exit**; plan mode — это режим агента, который пишет план в файл и завершает «планировочный» ход вызовом plan_exit; отдельной фазы «сначала план тулов» в коде нет.

**Где агент «думает» / планирует:**
- **Встроенный ризонинг:** в процессоре явно обрабатываются `reasoning-start`, `reasoning-delta`, `reasoning-end` — создаётся и обновляется **MessageV2.ReasoningPart**, сохраняется в сессию и отображается. Модели в API помечены флагом **`reasoning: true/false`** и полем **`interleaved: { field: "reasoning_content" }`** (или аналог) — т.е. ризонинг приходит в стриме и маппится в отдельный тип части сообщения. Связь «встроенный ризонинг модели» ↔ «reasoning parts в сообщении» прямая: провайдер отдаёт reasoning в стриме, процессор превращает их в части сообщения.
- **Планирование:** в промптах есть инструкции «develop a plan», «plan next steps»; в UI есть статусы «Planning next steps», «Thinking» — они выводятся по состоянию сессии/хода, а не по отдельному шагу «только план» в лупе. Plan mode реализован через тулы и смену контекста (plan file), не через отдельный подцикл планирования.

**Ризонинг в схемах:**
- StructuredOutput тул — схема для **финального ответа** (JSON по схеме), не для «схемы рассуждения». В тулах вроде todo есть текстовые инструкции про reasoning в .txt; в схемах параметров тулов отдельного поля reasoning нет (кроме текстовых полей типа ответа/плана).

### 1.4 Roo-Code

**Где:** `src/core/task/Task.ts` — цикл `while (!this.abort)` с вызовом **`recursivelyMakeClineRequests(nextUserContent, includeFileDetails)`**. Запросы идут в Cline-совместимый API (Anthropic и др.).

**Как устроена лупа:**
- В комментарии в Task явно сказано: Cline получает задачу, вызывает тулы; пока нет `attempt_completion`, мы продолжаем слать ему ответы с результатами тулов; если тулов не было — отправляем напоминание «consider if completed / use attempt_completion». То есть лупа — это **последовательность запросов к Cline-совместимому API** с контентом пользователя и результатами тулов; «агентная» логика живёт на стороне API/модели.
- Внутри `recursivelyMakeClineRequests` — стек запросов, определение протокола API по провайдеру/модели, формирование сообщений и вызов API. Выполнение тулов идёт на стороне Roo/Cline (или через общий с Cline контракт).

**Где агент «думает» / планирует:**
- **Встроенный ризонинг:** в `src/shared/api.ts` — **`shouldUseReasoningEffort`**, **`shouldUseReasoningBudget`**; учитываются `model.supportsReasoningEffort`, `model.requiredReasoningBudget`, `settings.reasoningEffort` (disable, none, minimal, low, medium, high). Эти настройки передаются в провайдер — т.е. управляют встроенным ризонингом модели в API, а не отдельным шагом в коде. В `src/utils/tag-matcher.ts` — парсинг тегов вроде `<think>...</think>
- **Планирование:** отдельной фазы планирования тулов в коде нет; поведение определяется промптами и API, как у Cline.

**Ризонинг в схемах:**
- В тулах есть параметры вроде `reason`, `mode_slug` (switch_mode); это не «схема рассуждения», а аргументы тулов. Встроенный ризонинг задаётся настройками провайдера, не полями в схемах тулов.

## 3. Ризонинг в схемах (тулзы, structured output)

- **Параметры тулов с полем «reasoning»:**  
  - У нас: только у **thinking_preamble** — поле `reasoning_and_next_actions` (текст рассуждений).  
  - В Cline/Kilocode/Roo в описаниях тулов есть текстовые инструкции «explain», «reason», но отдельного обязательного поля типа `reasoning: string` в JSON Schema тулов нет.  
  - В классификаторах NexusCode ответ может содержать `reasoning` — используется для отбора тулов/скиллов, не для шага «подумать» в лупе.

- **Structured output с полем reasoning:**  
  - У нас: нет режима «ответ по JSON Schema» с полем reasoning.  
  - В Kilocode: при `format.type === "json_schema"` добавляется тул StructuredOutput с произвольной схемой (в т.ч. теоретически с полем reasoning); в коде по умолчанию это не «схема рассуждения», а схема финального ответа.  
  - В Cline/Roo: отдельного structured output с полем reasoning в агентной логике не описано.

- **Итог:** ризонинг в схемах везде точечный: либо один тул с полем под рассуждение (у нас thinking_preamble), либо текстовая инструкция в промпте/описании тула. Отдельной «схемы рассуждения» как контракта агентной лупы нигде нет.

## 4. Связь с встроенным ризонингом модели

- **Общее:** во всех проектах «думание» в итоге опирается на то, что модель либо стримит отдельный reasoning/thinking, либо пишет рассуждение в текст/тул. Отдельного шага «сначала вызвать API только для reasoning, потом отдельно для действий» в лупе ни у кого нет.

- **NexusCode:** мы принимаем `reasoning_delta` из стрима и сохраняем в сообщении; при следующем запросе контент ассистента строится как reasoning → text → tool_calls. Мы **не** передаём в провайдер параметры вроде reasoning_effort или thinking_budget — их можно добавить по аналогии с Cline/Roo, если провайдер это поддерживает.

- **Cline:** для моделей с поддержкой (gpt, o, gemini, grok и т.д.) в API уходят reasoning effort и thinking budget; провайдеры возвращают reasoning в стриме, ReasoningHandler превращает их в thinking block. Связь «включён встроенный ризонинг в API» ↔ «в чате отображается thinking» прямая.

- **Kilocode/OpenCode:** в метаданных моделей помечено `reasoning: true` и способ экспозиции (напр. `interleaved: { field: "reasoning_content" }`); процессор обрабатывает reasoning-start/delta/end и создаёт ReasoningPart. Связь «модель с reasoning» ↔ «части сообщения типа reasoning» явная в коде процессора.

- **Roo-Code:** настройки reasoning effort и budget передаются в провайдер; парсинг тегов `<think>` через TagMatcher позволяет выделять рассуждение из текста стрима, если провайдер отдаёт его в таком виде. Связь — через параметры API и контракт стрима.

## 5. Выводы и возможные улучшения у нас

- **Лупа:** у нас одна итерация = один stream(); при tool_calls мы не делаем отдельного «планировочного» запроса — модель сама решает порядок вызовов и может использовать thinking_preamble. Это сопоставимо с Cline/Kilocode/Roo: везде один запрос даёт текст + (опционально) reasoning + tool_calls; следующий запрос уже с результатами тулов.

- **Где агент думает:** у нас — (1) нативный reasoning_delta, (2) явный вызов thinking_preamble. В других — нативный reasoning в стриме + в Cline ещё plan/act режимы и тулы для плана и прогресса. Отдельной фазы «сначала план тулов по схеме» ни у кого нет.

- **Связь со встроенным ризонингом:** мы пока не передаём в провайдер параметры reasoning effort / thinking budget. Имеет смысл по аналогии с Cline/Roo добавить опцию в конфиг и проброс в `stream()` для поддерживаемых провайдеров, чтобы модели типа o1/R1/Grok могли получать нужный объём thinking.

- **Ризонинг в схемах:** везде точечный (одно поле в одном туле или инструкция в промпте). Добавление отдельного «поля reasoning» в другие тулы или в единый контракт ответа — возможное расширение, но ни в одном из проектов так не сделано по умолчанию.

## 6. Есть ли в других проектах режим с дополнительными полями ризонинга?

**Краткий ответ: нет.** Ни в одном из рассмотренных проектов нет выделенного режима вида «включить доп. поля ризонинга во всех тулах» или «отдельный шаг только ризонинг по схеме».

**По проектам:**

- **Cline**  
  Нет отдельного режима «добавить поля ризонинга». Plan mode ограничивает набор тулов и даёт один тул для плана (`plan_mode_respond` с параметрами `response`, `needs_more_exploration`). Встроенный ризонинг идёт через стрим (thinking block), а не через доп. поля в схемах тулов. Настройки reasoning effort / thinking budget управляют только встроенным ризонингом модели в API.

- **Kilocode / OpenCode**  
  Ближе всего к «доп. полям» — **structured output по JSON Schema**: при `format.type === "json_schema"` можно задать схему с полем `reasoning` (или аналогом) в **финальном ответе** (тул StructuredOutput). Отдельного режима «режим ризонинга с полями во всех тулах» нет; рассуждение в сообщении — через нативный стрим (reasoning-start/delta/end → ReasoningPart).

- **Roo-Code**  
  Ризонинг управляется настройками провайдера (reasoning effort / budget) и парсингом тегов в стриме. Режима, который добавляет поля ризонинга в тулы или в отдельный шаг, нет.

**Итог:** возможность «доп. поля ризонинга в отдельном режиме» есть только в виде (1) финального ответа по схеме с возможным полем reasoning (Kilocode/OpenCode) и (2) нативного стрима рассуждений + опционально тулов вроде plan_mode_respond / thinking_preamble без отдельной «режимовой» инъекции полей ризонинга во все тулы.
